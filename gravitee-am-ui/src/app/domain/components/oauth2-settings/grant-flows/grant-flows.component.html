OAuth 2.0 / OpenID Connect

Grant types

To authenticate an user, the application must first decide which oauth2 flow must be used. Oauth2 flows are enabled on the client through the grant_type. You should select the grant_type according to your application type.

For MCP Resource Servers, only the {{ CLIENT_CREDENTIALS_GRANT_TYPE }} grant type and token exchange flows are supported. These grant types enable server-to-server authentication where the client authenticates using its credentials to obtain an access token.

authorization_code for server side applications (web) or browser and native applications (then combined with PKCE).
implicit originally used for browser apps, deprecated in favor of authorization_code + PKCE.
password should only be used for trusted applications (means application and authorization server are managed within the same organization).
{{ CLIENT_CREDENTIALS_GRANT_TYPE }} should be used for server to server communications.

It is not recommended to mix client and user authentication within the same application. (Aka {{ CLIENT_CREDENTIALS_GRANT_TYPE }} and authorization_code, implicit or password)
 {{ CLIENT_CREDENTIALS_GRANT_TYPE }} is used for server-to-server communications where the client authenticates using its credentials to obtain an access token.
Token exchange allows exchanging an existing token for a new token, enabling delegation and impersonation scenarios. Enable Token Exchange in domain settings if the option is disabled.

These grant types are appropriate for Resource Servers that need to authenticate and authorize client applications accessing protected resources.
